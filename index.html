Git이란?

Git은 2005년 리누스 토르발스(Linus Torvalds)가 리눅스 커널 개발을 위해 만든 분산 버전 관리 시스템(DVCS)입니다. 파일의 변경 내역을 스냅샷으로 기록하고, 언제든 이전 상태로 되돌리거나 변경 사항을 비교할 수 있습니다.

📸

스냅샷 기록

프로젝트의 특정 시점을 스냅샷으로 저장하여 변경 이력을 완벽하게 추적합니다.

🌐

분산 구조

모든 개발자가 전체 저장소 복사본을 가지므로, 중앙 서버 없이도 작업이 가능합니다.

🔀

브랜치 & 병합

가볍고 빠른 브랜치 생성으로 기능 개발, 버그 수정을 독립적으로 진행할 수 있습니다.

Git vs GitHub — Git은 버전 관리 도구이고, GitHub는 Git 저장소를 호스팅하는 웹 서비스입니다. Git 없이 GitHub를, GitHub 없이 Git을 사용할 수 있습니다.

왜 Git을 쓸까?

상황 Git 없이 Git 사용 시
코드 수정 후 문제 발생 백업 파일을 찾아 수동 복원 git checkout으로 즉시 이전 버전 복원
팀원과 동시 작업 파일 덮어쓰기, 충돌 빈번 브랜치로 독립 개발 후 병합
누가 무엇을 바꿨는지 기억에 의존 git log, git blame으로 추적
실험적 기능 시도 원본 파일 복사 후 작업 브랜치 생성 → 실험 → 병합 또는 삭제

초기 설정

Git을 처음 사용하기 전에 사용자 이름과 이메일을 설정합니다. 이 정보는 모든 커밋에 기록됩니다.

Terminal
git config --global user.name "홍길동"
git config --global user.email "hong@example.com"

# 설정 확인
git config --list
기본 에디터 설정 — 커밋 메시지 작성 시 사용할 에디터를 지정할 수 있습니다: git config --global core.editor "code --wait" (VS Code)

기본 명령어

일상적으로 가장 많이 사용하는 Git 명령어들입니다.

명령어 설명
git init 현재 폴더를 Git 저장소로 초기화
git clone <url> 원격 저장소를 로컬로 복제
git status 작업 트리 상태 확인 (변경/스테이징 파일)
git add <file> 변경 파일을 스테이징 영역에 추가
git add . 모든 변경 파일을 스테이징
git commit -m "메시지" 스테이징된 변경 사항을 커밋(저장)
git log 커밋 이력 조회
git diff 변경 내용 비교
git restore <file> 파일의 변경 사항 되돌리기

저장소 만들기 & 첫 커밋

Terminal
# 새 프로젝트 시작
mkdir my-project && cd my-project
git init

# 파일 작성 후
echo "# My Project" > README.md
git add README.md
git commit -m "Initial commit: README 추가"

기본 워크플로우

Git에서 파일은 세 가지 영역을 거칩니다. 이 흐름을 이해하면 Git 사용이 훨씬 수월해집니다.

Working Directory
→
git add
→
Staging Area
→
git commit
→
Repository

Working Directory

실제 파일을 수정하는 작업 공간입니다. 변경된 파일은 "modified" 상태가 됩니다.

Staging Area

git add로 다음 커밋에 포함할 변경 사항을 선택하는 영역입니다.

Repository

git commit으로 영구적으로 기록된 스냅샷이 저장되는 곳입니다.

.gitignore

커밋하지 않을 파일/폴더를 지정합니다. 빌드 결과물, 환경 변수 파일, OS 생성 파일 등을 제외할 때 사용합니다.

.gitignore
# 의존성
node_modules/
__pycache__/

# 환경 변수
.env
.env.local

# OS 파일
.DS_Store
Thumbs.db

# 빌드 결과물
dist/
build/

브랜치 & 병합

브랜치는 독립적인 개발 라인을 만드는 Git의 핵심 기능입니다. 새 기능 개발이나 버그 수정 시 main 브랜치를 건드리지 않고 안전하게 작업할 수 있습니다.

Terminal
# 브랜치 목록 확인
git branch

# 새 브랜치 생성 & 이동
git checkout -b feature/login
# 또는 (Git 2.23+)
git switch -c feature/login

# 작업 후 커밋
git add .
git commit -m "로그인 기능 추가"

# main으로 돌아가 병합
git switch main
git merge feature/login

# 완료된 브랜치 삭제
git branch -d feature/login
충돌(Conflict) — 두 브랜치가 같은 파일의 같은 부분을 수정하면 병합 시 충돌이 발생합니다. Git이 충돌 지점을 표시해 주므로, 파일을 직접 수정한 뒤 git add → git commit으로 해결합니다.

원격 저장소

GitHub, GitLab 등의 원격 저장소와 연동하여 코드를 공유하고 협업합니다.

Terminal
# 원격 저장소 연결
git remote add origin https://github.com/user/repo.git

# 로컬 커밋을 원격에 업로드
git push -u origin main

# 원격의 최신 변경 사항 가져오기
git pull origin main

# 원격 변경 사항 확인 (병합 없이)
git fetch origin

협업 워크플로우

git pull
→
브랜치 생성
→
작업 & 커밋
→
git push
→
Pull Request
명령어 설명
git remote -v 연결된 원격 저장소 목록 확인
git push 로컬 커밋을 원격 저장소에 업로드
git pull 원격 변경 사항을 가져와 병합 (fetch + merge)
git fetch 원격 변경 사항을 가져오기만 함 (병합 안 함)
git clone <url> 원격 저장소 전체를 로컬로 복제

유용한 팁

자주 쓰는 단축

Terminal
# 마지막 커밋 메시지 수정
git commit --amend

# 스테이징 취소
git restore --staged <file>

# 커밋 이력을 보기 좋게
git log --oneline --graph --all

# 특정 파일의 변경 이력
git log --follow -p <file>

# 임시 저장 (스태시)
git stash
git stash pop

fix: 모바일에서 레이아웃 깨지는 문제 수정

❌ 나쁜 예

수정
asdfasdf
버그 수정 및 기능 추가 및 리팩토링

Conventional Commits — feat:, fix:, docs:, refactor: 등 접두사를 사용하면 커밋 이력이 한눈에 파악됩니다.
